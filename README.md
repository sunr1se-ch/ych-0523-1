# 社区旧书漂流柜流转登记系统

用于管理社区共享书柜借阅流转的全栈应用，支持柜格管理、借还登记、逾期提醒、待清柜处理等功能。

## ✨ 功能特性

### 核心功能
- **柜格总览**：可视化展示所有柜格状态（空位/在借/已满/待清柜），实时统计数据
- **单格详情**：查看指定柜格的借阅时间线、历史记录、当前状态
- **借书登记**：登记居民姓名、书名、取走时间、预计归还日
- **还书登记**：登记实际归还日、记录书脊磨损等级（1-5级）
- **待清柜管理**：查看待清柜列表，执行清柜确认操作
- **逾期提醒**：逾期图书列表，按逾期天数排序，支持快捷还书

### 业务规则
- **容量限制**：每个柜格有固定容量上限，在借册数达上限时禁止新借书
- **逾期触发**：任一册图书逾期超过 3 天未还，柜格自动进入「待清柜」状态
- **待清柜限制**：待清柜期间禁止新借书登记，但仍接受还书
- **清柜确认**：管理员手动确认清柜后，柜格恢复正常使用

## 📋 待清柜期间强行还书处理规则

### 规则说明

当柜格处于「待清柜」状态时，**仍可正常接受还书登记**，但需遵循以下规则：

| 操作类型 | 是否允许 | 系统行为 |
|---------|---------|---------|
| **借书登记** | ❌ 禁止 | 提示「该柜格处于待清柜状态，禁止借书，请先完成清柜」 |
| **还书登记** | ✅ 允许 | 正常登记还书信息，记录书脊磨损等级，更新在借册数 |
| **清柜确认** | ✅ 允许 | 管理员手动操作，确认后柜格恢复正常状态 |

### 详细处理流程

1. **强行还书触发**：待清柜期间，居民前来归还逾期图书
2. **系统登记**：管理员正常执行还书操作，系统：
   - 记录实际归还日期
   - 记录书脊磨损等级（1-5）
   - 该柜格在借册数 -1
3. **状态保持**：**柜格不会自动解除待清柜状态**，仍保持「待清柜」
4. **管理员清柜**：管理员需在「待清柜管理」页面手动点击「确认清柜」
5. **状态恢复**：确认清柜后，系统重新评估柜格状态：
   - 如仍有其他逾期超过 3 天的图书 → 保持待清柜
   - 如无其他超期图书 → 根据当前在借册数恢复为「空位/在借/已满」

### 设计原因

此规则的设计考虑了以下场景：
- 居民可能在收到逾期提醒后主动还书，无需管理员强制清柜
- 保证每一笔还书都有完整记录（磨损等级、归还时间）
- 管理员仍保有最终控制权，确保柜格经过清点后再开放使用
- 避免因个别图书归还而自动恢复柜格，导致其他逾期图书被忽略

## 🔌 API 接口

### 柜格管理

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/stats` | 获取统计数据 |
| GET | `/api/cabinets` | 获取所有柜格列表 |
| GET | `/api/cabinets/:id` | 获取单个柜格详情（含记录） |
| GET | `/api/cabinets/:id/records` | 获取柜格借阅记录 |

### 借还管理

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | `/api/borrow` | 借书登记 |
| POST | `/api/return/:id` | 还书登记 |
| POST | `/api/cleanup/:id` | 确认清柜 |

### 提醒与待办

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/overdue` | 获取逾期借阅列表 |
| GET | `/api/pending-cleanup` | 获取待清柜列表 |

### 数据模型

#### 柜格状态枚举
- `available` - 空位（0册在借）
- `partial` - 在借中（1~容量-1册在借）
- `full` - 已满（达到容量上限）
- `pending_cleanup` - 待清柜

#### 书脊磨损等级
- **1级**：崭新，无磨损
- **2级**：轻微磨损，书脊完整
- **3级**：中度磨损，有折痕但不影响阅读
- **4级**：严重磨损，书脊有开裂
- **5级**：损坏严重，需下架维修
